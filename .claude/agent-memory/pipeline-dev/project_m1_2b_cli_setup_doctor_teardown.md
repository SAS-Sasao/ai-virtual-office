M1-2b（`feat/2026-07-21-m1-2b-setup-cli` ブランチ、`packages/cli/` のみ担当。`packages/relay` の `/health` 拡張は別担当が同一ブランチ上で並行作業していた。設計メモ rev.3（人間承認済み）に厳密に従い TDD で実装。初回実装は 85 テスト green だったが、**Phase3 レビュー（composite 0.81 fail、9 件すべて実測で再現）で高 1 件・中 6 件・低 2 件の指摘を受け、TDD で修正して 106 テスト green まで積み増した**（詳細は本メモ後半）。typecheck/build とも exit 0。`npm pack` → クリーン一時ディレクトリで `npm i` → `npx ai-office --help` を実測して exit 0 かつ stdout にヘルプ本文が出力されることを確認（AC-10、Phase3 で「exit code だけ見ていた」検証不足も指摘され、stdout 内容の assertion を追加した）。作業前後で本物の `~/.claude/settings.json` / `settings.local.json` の sha256 が完全不変であることを確認。関連: [[m1-2b-relay-health-stats]] [[m1-1-relay-creation]]

**`npx`/`.bin` シンボリックリンク経由だと `isMainModule` 判定が silently false になる罠**: `import.meta.url === \`file://${process.argv[1]}\`` という単純比較は、`node dist/index.js` の直接実行では動くが、`npx ai-office`（= `node_modules/.bin/ai-office` というシンボリックリンク経由）では `process.argv[1]` がリンクパス、`import.meta.url` は解決済みの実体パスになり**一致しない**。結果、トップレベルの起動ガードが false になり、**ヘルプすら表示せず exit 0 で無言終了する**（エラーにすらならないため見つけにくい）。`fs.realpathSync(process.argv[1])` と `fileURLToPath(import.meta.url)` を比較する形に直して解決した。**bin を持つ Node CLI パッケージを作るときは、`node dist/x.js` の直接実行だけでなく、必ず `npm pack` → 別ディレクトリで `npm i` → `npx` 経由の実行まで確認すること**（ローカルの直接実行では絶対に再現しない）。

**（初回実装 → Phase3 で撤回）マーカーの「接頭辞」除外は二重送信を再導入する**: 初回実装では重複検出を `"#ai-office"` 接頭辞の有無で行っていたが、これは**本リポジトリ形状（手書き `#ai-office`・同一ポート）に `setup` すると全 8 イベントが 2 本立ちし全イベント二重送信になる**という致命的な回帰を生んでいた（Phase3 finding3、実測で再現）。手書き `#ai-office` を「意図的に管理されている」とみなして除外したのが誤り。**正しい判定は「同一 URL を叩く既存 hook があり、かつそれが完全体マーカー `#ai-office:cli` を持たない」の一本**（`h.command.includes(MARKER)` の否定）。手書き `#ai-office` も他と同様「マーカー無し」として重複検出にかかり、追加しない。AC-1（既存を壊さない）はこの判定と両立しない場合があるため、**AC-1 のテストは元 fixture と別ポート、finding3 の再現テストは元 fixture と同一ポートで検証**し、2つのシナリオを明確に分離した。

**（初回実装 → Phase3 で撤回）`removeHooks` は `settings.hooks` 自体を削除してよい（条件付き）**: 初回実装は「`hooks` キーは元から存在していたので絶対に削除しない」という原則で実装したが、これは**事実誤認**だった。実物の `~/.claude/settings.json` は `hooks` キー自体を持たない（model/enabledPlugins/effortLevel/theme のみ）。`mergeHooks` は `hooks` キーが無ければ新設するため、そこから `setup → teardown` を round-trip すると、`hooks` キーを保持したままだと `"hooks": {}` が残置され AC-3 の deep-equal が崩れる（Phase3 finding2、実測で再現）。**正しい実装**: 除去の結果 `hooksObj` が空オブジェクトかつ `removedCount > 0`（今回何かを実際に除去した）なら `hooks` キーごと削除する。**トレードオフ**: 元々 `hooks: {}` を明示的に持っていたファイルではこのキーが消える（`{}` → キー無し）が、Claude Code は両者を等価に扱うため実害なしとして許容（人間レビュー承認済み）。3パターン（hooks キー無し / 空 `{}` / 他ツール hooks 同居）の round-trip テストで固定した。

**`mergeHooks` が既存の空グループに追記すると `removeHooks` がそれを丸ごと消してしまう（finding9）**: `{matcher:"*", hooks: []}` のような**元から存在する空グループ**に `mergeHooks` が追記すると、後で `removeHooks` は「このグループは中身が全部 CLI 由来 → 空になった → 畳んで削除」と判断してしまい、**グループ自体（元々あった空の器）まで消える**。この種の曖昧さ（「このグループは元から空だったのか、CLI が作ったばかりなのか」）は `removeHooks` 単体では原理的に判別不能。**根本解決は `mergeHooks` 側**: 一致する既存グループのうち「中身が入っている（`hooks.length > 0`）」ものにのみ追記し、一致するが空のグループには**触れず**、新規グループを別途作成する。これにより `removeHooks` が空にできるグループは常に「完全に CLI が新規作成したもの」に限定され、`removeHooks` 側のロジックは無変更のまま安全になった（実際に repro スクリプトで `PreToolUse` キーが `undefined` になることを確認してから直した）。

**teardown の「バックアップも残らない」(AC-3) は teardown 自身のバックアップ削除だけでは満たせない**: `writeSettingsSafely` は setup・teardown 双方の書き込みで「既存ファイルがあれば必ずバックアップを取る」ため、**setup 自体の書き込みでもバックアップファイルが 1 個作られ、setup はそれを削除しない**（設計メモが明示的に削除を求めているのは teardown の分のみ）。**（初回実装 → Phase3 で絞り込み）** 最初は `${realPath}.ai-office-backup-*` に一致する全件を無差別に削除していたが、これは**過去に `--keep-backup` で利用者が意図的に残したバックアップまで消してしまう**指摘を受けた（finding7）。**正しい実装**: 削除対象を「今回 teardown 自身が作った分」＋「それを除いた中でファイル名の ISO8601 タイムスタンプが最も新しい1件（＝直前の setup が残した分と推定できるもの）」の**最大2件**に限定する。それより古いものには一切触れない。

**relay の内部定数を「変更せずに」ドリフト検知する**: AC-11c（slug の二重管理防止）は `devDependencies` に `@ai-office/relay`（workspace:*）を入れて検証する設計だったが、`HOOK_EVENT_SLUGS` は `packages/relay/src/server.ts`内の非 export 定数で、かつ relay の `package.json` に `main`/`exports` も無い（`bin` のみ）。relay は別担当が並行作業中で触れない制約があったため、**relay 側を一切変更せず**、`createRequire(import.meta.url).resolve("@ai-office/relay/package.json")` でワークスペースの実体ディレクトリを解決し、`src/server.ts` の**ソーステキストを直接読んで正規表現で `HOOK_EVENT_SLUGS = new Set([...])` を抽出**する形でドリフト検知テストを書いた（dist ビルド不要、relay の export 方針にも依存しない）。relay 側が将来この定数を export するようになったら、正規表現抽出は不要になるので置き換えてよい。

**`dependencies: {}` を維持しつつ devDependency で workspace パッケージを参照する構成は npm pack と両立する**: `packages/cli/package.json` は `dependencies: {}` / `devDependencies` に `@ai-office/relay: workspace:*` を置いた。`npm pack` の tarball には `package.json` と `dist/` しか含まれず（`files: ["dist"]`）、devDependencies は同梱されないため配布物に影響しない。クリーンな一時ディレクトリで `npm i <tgz>` した際も `added 1 package` のみで workspace 依存が一切引き込まれないことを実測確認済み。

## Phase3 レビューで踏んだ追加の落とし穴（finding1・4・5・8）

**「既存値が期待形状（配列）でない」を無警告で上書き削除していた（finding1・high）**: 初回実装は `hooksObj[event]` や `group.hooks` が配列でない場合、`Array.isArray(...) ? ... : []` の三項演算子で**無条件に空配列へ差し替えていた**。これは「既存が壊れている/未知形状」と「既存が単に存在しない」を区別しておらず、レビュアーの実測で `PostToolUse: "my-precious-guard.sh"` のような壊れた設定が**無警告で消滅**した。**修正**: `groupsRaw !== undefined && !Array.isArray(groupsRaw)` を明示的に検出し、その場合は一切書き換えず `skippedMalformedSlugs` に積んでそのイベントをスキップする（`MergeResult` に新フィールド追加）。「値が無い（undefined）ときだけ新設してよい」と「値はあるが形が違うときは触らない」を明確に分けるのが鉄則。`setup.ts` はこれを警告メッセージとして表示する。

**doctor と setup の判定基準がズレて相互矛盾する（finding4）**: マーカー無し手書き hook で実際には配線が成立している slug（`skippedDuplicateSlugs`）を、setup は「追加不要」として exit 0 を返すのに、doctor の `hooksInstalled` 判定は `installedSlugs`（＝完全体マーカー付き）だけを見ていたため 7/8 のまま exit 1 になり、**setup 成功直後に doctor が fail する**という利用者から見て矛盾した挙動になっていた。**修正**: `hooksInstalled` の判定を `installedSlugs.length + duplicateSlugs.length === HOOKS_SPEC.length` に変更（「マーカー付きで正式管理」と「マーカー無しの手書きだが URL は届いている」の両方を"配線成立"としてカウント）。**教訓**: 複数コマンド（setup/doctor）が同じ状態を別の切り口で判定するときは、判定条件を1箇所（`mergeHooks` の分類結果）に統一しないと簡単に矛盾する。

**exit code だけを見るテストは npx 経由のサイレント無出力バグを検出できない（finding5）**: 前回セッションで `isMainModule` の realpath 比較バグを直したが、そのときのテスト（AC-10 実測）は「`npx ai-office --help` の exit code が 0」しか見ていなかった。バグ版（`import.meta.url === file://${argv[1]}` の単純比較）も**`.bin` シンボリックリンク経由では exit 0 のまま stdout が空文字列**になるため、exit code ベースの検証ではこのバグを再現・防止できない。**修正**: `execFileSync` で実子プロセスとして `.bin` 相当のシンボリックリンクを張って起動し、**stdout の中身（HELP_TEXT と完全一致）まで assert する** `cli-entrypoint.test.ts` を追加。実際にバグ版へ一時的にロールバックして「stdout が空になり2テストが red になる」ことを確認してから正しい実装に戻す、という mutation-testing 的な手順で有効性を検証した。**教訓**: bin エントリポイントのテストは exit code だけでなく出力内容まで見ないと、"静かに失敗する" 系のバグを取りこぼす。

**fs の生例外を分類しないと未捕捉スタックトレースでプロセスが落ちる（finding8）**: `loadSettings` 内の `readFileSync`/`lstatSync` は元々 try/catch 無しで、対象パスがディレクトリ（EISDIR）等の場合に生の Node エラーがそのまま `setup.ts`/`teardown.ts` の `catch (err) { if (err instanceof SettingsParseError) {...}; throw err; }` を素通りして**未捕捉例外として上位に伝播**していた。**修正**: 新設した `SettingsReadError`（`SettingsParseError`/`SettingsWriteError` と同系統の型付きエラー）で読み取り失敗を包み、`setup.ts`/`teardown.ts` 側の catch 節にも追加。加えて `index.ts` 末尾の `runCli(...).then()` に `.catch()` を付け、それでも漏れた想定外の例外は「1行メッセージ + exit 1」で確実に収める最後の安全網とした。`runCli` 自身が内部で例外を握り潰さない（＝ `.catch()` が意味を持つ前提）ことを `expect(runCli(...)).rejects.toThrow(...)` で固定するテストも追加した。
